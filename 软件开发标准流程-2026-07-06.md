## 一、项目定位

ShopLite 是一个以 Django 为核心的电商网站项目，当前采用服务端模板渲染为主、少量 JSON API 辅助前端交互的架构。项目已经覆盖商城核心链路：

- 用户登录、注册、手机号验证码登录、mock 微信登录。
- 商品分类、商品列表、搜索建议、商品详情。
- 购物车、下单、订单列表、订单详情。
- 模拟支付、支付回调、支付后扣库存和增加销量。
- 收货地址管理、收藏管理。
- Django Admin 后台管理。
- Celery + Redis 定时取消超时待付款订单。
- Docker Compose 部署 MySQL、Redis、Web、Celery、Nginx。

项目不是前后端分离应用，也不是 DRF/JWT 架构。当前主要页面由 Django templates 输出，前端 JavaScript 只负责搜索下拉、筛选跳转、登录 API 调用、商品详情交互等局部增强。

## 四、核心业务模块

### 1. 用户与认证

相关文件：

- `shop/services.py`
- `shop/api_views.py`
- `templates/account/login.html`
- `templates/account/signup.html`
- `templates/account/logout.html`
- `templates/account/password_reset.html`
- `shop/models.py` 中的 `UserProfile`、`PhoneVerificationCode`

当前能力：

- allauth 用户名密码登录/注册页面。
- 自定义 API 支持手机号验证码注册。
- 自定义 API 支持手机号验证码登录。
- 自定义 API 支持手机号或用户名密码登录。
- mock 微信登录会创建或复用 allauth `SocialAccount`。
- 手机号绑定服务函数已存在。

开发标准：

- 登录相关业务规则统一放在 `shop/services.py`。
- JSON 接口放在 `shop/api_views.py`。
- 账号页面模板放在 `templates/account/`。
- 验证码默认打印在控制台，生产环境必须切换真实短信服务。
- 手机号必须通过 `normalize_phone()` 校验，不允许视图中重复写正则逻辑。

待完善：

- `password_reset.html` 当前是静态设计稿，没有真正接后端重置流程。
- `signup.html` 中手机号字段未真正纳入 allauth 注册保存逻辑，手机号注册以登录页的“手机注册”Tab 为准。
- 账户设置页修改手机号/邮箱/昵称只是前端提示，未持久化。

相关文件：

- `shop/models.py`：`Category`、`Product`、`ProductImage`、`Review`

当前能力：

开发标准：

- 购物车页支持删除。
- API 支持添加、更新、删除。

开发标准：

- 不在视图中直接写购物车金额计算，统一用 `calculate_cart_totals()`。
- 添加购物车统一用 `add_product_to_cart()`。
- 数量解析统一用 `parse_quantity()`。
- 顶部购物车预览统一来自 `base_context()`。

待完善：

- `cart.html` 中数量加减按钮目前只是 UI，没有绑定 `cart_update` API。
- `CartItem.is_selected` 已有字段，但前台没有完整选中/取消选中结算逻辑。

### 4. 订单

相关文件：

- `shop/models.py`：`Order`、`OrderItem`
- `shop/services.py`：`create_order_from_cart()`、`mark_order_paid()`、`cancel_pending_order()`
- `shop/views.py`：`checkout()`、`place_order()`、`order_list()`、`order_detail()`
- `shop/api_views.py`：`order_create()`
- `shop/templates/checkout.html`
- `shop/templates/order_list.html`
- `shop/templates/order_detail.html`
- `shop/tasks.py`

当前能力：

- 从购物车创建订单。
- 订单包含商品明细快照。
- 订单包含地址快照。
- 待付款订单不扣库存。
- 支付成功后扣库存、加销量。
- 重复支付不会重复扣库存。
- 可按订单状态筛选列表。
- Celery 定时取消超时待付款订单。

订单状态：

- `pending`：待付款。
- `paid`：待发货。
- `shipped`：待收货。
- `completed`：已完成。
- `cancelled`：已取消。
- `refund`：退款/售后。

开发标准：

- 创建订单必须使用 `create_order_from_cart()`。
- 支付成功必须使用 `mark_order_paid()`。
- 涉及订单状态和库存必须放在事务中处理。
- 不允许在模板或视图中直接扣库存。
- 订单商品必须保存快照，不能只依赖商品外键。

待完善：

- 没有前台确认收货、申请售后、取消订单按钮。
- 没有真实物流信息。
- 没有订单状态机封装，状态流转主要靠服务函数和后台操作。

相关文件：

- `shop/payments.py`

- `shop/services.py`：`mark_order_paid()`
- `shop/templates/order_detail.html`

当前能力：

- mock 支付页面直接标记订单已支付。
- 支付通知支持 JSON 或表单 payload。
- 可配置 `SHOPLITE_PAYMENT_SECRET` 做 HMAC 签名校验。
- 校验支付金额必须等于订单应付金额。
- 支付成功后调用 `mark_order_paid()`。

开发标准：

- 支付通知必须幂等。
- 支付金额必须校验。
- 支付状态必须校验。
- 支付成功后的库存扣减必须走服务层。
- 如果接入真实支付，不要绕过 `handle_payment_notification()` 的核心校验思路。

待完善：

- 当前不是真实支付宝或微信支付。
- `alipay_notify()` 只是复用 mock 通知处理，未做支付宝官方验签。
- `create_checkout_session()` 当前只是跳 mock payment，不是真支付会话。

## 五、分层开发规范

### 模型层：`models.py`

职责：

- 定义数据库结构。
- 定义简单属性，如 `subtotal`、`savings`、`is_expired`。
- 定义 Meta、排序、唯一约束。

禁止：

- 在模型中写复杂视图逻辑。
- 在模型中直接处理 request。

### 查询层：`selectors.py`

职责：

- 封装读取逻辑。
- 封装列表筛选。
- 封装页面公共上下文。
- 封装统计数据。

适合放：

- 商品列表筛选。
- 分类树。
- 品牌统计。
- 购物车数量。
- 订单状态统计。
- 评价统计。

### 服务层：`services.py`

职责：

- 处理会修改数据库的业务逻辑。
- 处理事务。
- 处理库存、订单、验证码、注册等规则。

适合放：

- 加购物车。
- 创建订单。
- 支付成功。
- 取消订单。
- 注册用户。
- 绑定手机号。
- 验证短信码。

必须遵守：

- 钱用 Decimal。
- 库存扣减用事务和 F 表达式。
- 订单支付必须幂等。
- 用户输入必须先规范化和校验。

### 页面视图：`views.py`

职责：

- 接收浏览器请求。
- 调用 selectors/services。
- 组织模板上下文。
- 返回 HTML 或跳转。

禁止：

- 在视图中直接写复杂业务规则。
- 在视图中直接拼大段 SQL 或复杂 ORM。

### API 视图：`api_views.py`

职责：

- 提供 JSON API。
- 给登录页、搜索建议、购物车等前端交互使用。

标准：

- 请求体用 `_json_payload()` 安全解析。
- 错误用 JSON 返回。
- 登录后操作加 `@login_required`。
- 修改类接口加 `@require_POST`。

### 模板层

职责：

- 展示数据。
- 少量页面交互。

标准：

- 公共布局放 `base.html`。
- 页面专属 CSS 可以放模板 `extra_head`，但重复样式应逐步沉淀到 `app.css`。
- 重复组件应考虑抽 include。
- 模板中不要写复杂业务判断。

## 六、标准开发流程

### 阶段 1：明确需求

每个功能开发前先确认：

- 是完整后端功能，还是展示型页面。
- 影响哪些模型。
- 是否需要数据库迁移。
- 是否影响订单、库存、支付等高风险链路。
- 是否需要登录。
- 是否需要后台管理。

输出：

- 需求说明。
- 涉及文件列表。
- 数据变更说明。

### 阶段 2：设计数据模型

如果需要新数据，先改 `models.py`。

标准步骤：

1. 定义模型字段。
2. 定义 verbose_name。
3. 定义排序和唯一约束。
4. 考虑索引。
5. 生成迁移。
6. 检查迁移文件。

命令：

```bash
python manage.py makemigrations
python manage.py migrate
```

### 阶段 3：补服务层和查询层

读取逻辑放 `selectors.py`。  
写入逻辑放 `services.py`。

标准：

- 函数名要表达业务动作。
- 错误使用 `ValidationError`。
- 需要一致性的操作加 `transaction.atomic`。
- 涉及库存用 `select_for_update()` 或 F 表达式。

### 阶段 4：接视图和路由

页面功能：

- 改 `views.py`。
- 改 `shop/urls.py`。
- 渲染模板。

API 功能：

- 改 `api_views.py`。
- 改 `api_urls.py`。
- 返回 JSON。

### 阶段 5：写模板和前端交互

页面开发顺序：

1. 先让 HTML 展示后端数据。
2. 再补 CSS。
3. 再补 JavaScript。
4. 最后检查空状态、错误状态、未登录状态。

### 阶段 6：后台管理

新模型需要考虑是否注册到 `admin.py`。

后台标准：

- `list_display`
- `list_filter`
- `search_fields`
- `readonly_fields`
- `list_select_related`
- 必要的批量 action

### 阶段 7：测试

重点测试：

- 订单金额。
- 库存扣减。
- 支付幂等。
- 验证码一次性。
- 手机号重复注册。
- 未登录访问。
- 用户只能操作自己的地址、订单、收藏。

当前测试入口：

```bash
python manage.py test shop
```

### 阶段 8：本地运行检查

开发运行：

```bash
cd shoplite
python manage.py runserver
```

Docker 运行：

```bash
cd shoplite
docker compose up --build
```

检查地址：

- 商城首页：`http://127.0.0.1:8000/`
- 后台：`http://127.0.0.1:8000/admin/`
- Docker + Nginx：`http://localhost/`

## 七、部署标准流程

### 1. 环境变量

从 `.env.example` 复制：

```bash
cp .env.example .env
```

生产环境必须设置：

- `DEBUG=False`
- `ALLOWED_HOSTS`
- `SECRET_KEY`
- `DATABASE_URL` 或数据库相关变量
- `SHOPLITE_PAYMENT_SECRET`
- 短信服务参数
- 微信登录参数

### 2. Docker Compose 服务

当前 `docker-compose.yml` 包含：

- `db`：MySQL。
- `redis`：Redis。
- `web`：Gunicorn + Django。
- `celery`：异步任务 worker。
- `celery-beat`：定时任务调度。
- `nginx`：反向代理和静态/媒体文件服务。

### 3. 静态文件和媒体文件

当前 Nginx 配置：

- `/static/` -> `/app/static/`
- `/media/` -> `/app/media/`

生产环境建议：

- 执行 `collectstatic`。
- 明确静态文件目录是 `static` 还是 `staticfiles`。
- 媒体文件使用持久化 volume 或对象存储。

### 4. 数据库迁移

容器启动后需要执行：

```bash
docker compose exec web python manage.py migrate
docker compose exec web python manage.py createsuperuser
```

如需收集静态文件：

```bash
docker compose exec web python manage.py collectstatic --noinput
```

## 八、安全标准

### 必须保留

- CSRF 中间件。
- 登录要求装饰器。
- 支付金额校验。
- 支付幂等。
- 订单和地址按当前用户过滤。
- 验证码有效期和发送频率限制。

### 生产环境必须调整

- `DEBUG=False`
- 不能使用默认 `SECRET_KEY`
- 不能使用默认数据库密码
- 配置真实 `ALLOWED_HOSTS`
- 配置 HTTPS
- 配置真实短信服务
- 如接真实支付，必须接官方验签

### 当前风险点

- `settings.py` 中 SimpleUI 配置存在乱码，需检查 Python 语法是否正常。
- 数据库密码默认值写在配置中，只适合本地开发。
- `password_reset.html` 没有真实重置逻辑。
- 地址删除使用 GET，不符合安全语义。

## 十五、最终标准

以后这个项目的开发判断，以当前代码结构为准：

- 不是 DRF/JWT 项目。
- 不是 PostgreSQL 项目。
- 不是 Redis 购物车项目。
- 不是多 app 拆分项目。
- 不是已接入真实支付宝/微信支付的项目。
- 是 Django 4.2 + MySQL + 服务端模板 + allauth + Celery/Redis + mock 支付的 ShopLite 商城项目。

所有新需求都应该沿着现有结构小步演进：

```text
models -> selectors/services -> views/api_views -> urls -> templates/static -> admin -> tests
```

这样项目会越来越一致，而不是变成多套架构混在一起。
